从输入url到页面加载完成发生了什么？

  • 简单描述:

    1、浏览器的地址栏输入URL并按下回车。
    2、浏览器查找当前URL是否存在缓存，并比较缓存是否过期。
    3、DNS解析URL对应的IP。
    4、根据IP建立TCP连接（三次握手）。
    5、HTTP发起请求。
    6、服务器处理请求，浏览器接收HTTP响应。
    7、渲染页面，构建DOM树。
    8、关闭TCP连接（四次挥手）。

  • 详细描述:

    1.准备动作: 现代浏览器会在你输入的过程中就开始智能匹配 URL，它会从书签、历史记录等地方，找到已经输入的字符串可能匹配到的 URL，然后智能显示出来，让你补全 URL。对于 Google Chrome，它甚至可以从缓存中把页面展示出来，也就是说，此时你还没有按下 Enter 进行搜索，页面就已经显示出来了。

正则

  • 18位身份证 :

    /^[1-9]\d{5}[1-9]\d{3}((0[1-9])|(1[0-2]))(([0[1-9])|([1|2]\d)|3[0-1])\d{3}[0-9Xx]$/

    6位地区编码 : [1-9]\d{5}

    4位年份 : [1-9]\d{3}

    2位月份 : (0[1-9])|(1[0-2])

    2位日数 : (([0[1-9])|([1|2]\d)|3[0-1])

    4位校验码 : \d{3}[0-9Xx]

  • regexp.exec(str),正则对象的主要方法,接收一个字符串参数,返回的数组,存放匹配的结果,[‘e’,index:1,input….等等]

    ​ 数组里有4个属性: length、index、input（被检索的string）、groups(undefined),未找到匹配则返回null

    ​ exec() 与match()类似，这个方法只会执行一次，加上修饰符 g 也只会执行一次，当再次调用的时候会从上次查询到的位置继续往后查。

  • regexp.test(str) , 检测一个字符串是否匹配某个模式,在全局匹配模式下,对于同一个正则对象重复调用就会出现下一次的匹配位置从上一次匹配结束的位置开始, 解决方法 : 重置lastIndex为0

  • str.match(regexp)是支持正则表达式的**String对象的方法 **,regexp 必需。规定要匹配的模式的 RegExp 对象。 如果该参数不是 RegExp 对象，则需要首先把它传递给 RegExp 构造函数，将其转换为 RegExp 对象。返回值为存放匹配结果的数组

  • str=str.replace(/(a)\1/g,'$1') 用第一个分组的值a对匹配到的值aa进行替换

  • 被正则表达式捕获(匹配)到的字符串会被暂存起来，分组1 : RegExp.$1

  • \1 和 '$1' 表示 第一个分组的值 括号就是分组

  • \d{1,3} 匹配1到3位的数字,等效于[0-9]

  • \D 查找非数字字符,等效于 [^0-9]

  • \w 匹配字母、数字、下划线,等效于[ A-Za-z0-9_ ]

  • \W 匹配非字母、数字、下划线

  • \b 匹配单词边界,比如er\b,可以匹配never却不可以匹配verb

  • \B 匹配非单词边界

  • \s 匹配空白字符,与\f,\n,\r,\t,\v等效

  • n+ 至少一个 n

  • n* 零个或多个 n

  • n? 零个或一个 n

  • /at/g 全局模式 ,匹配字符串中的所有at,而不是找到第一个就停下

  • /[bc]at/i 匹配第一个bat或者cat,不区分大小写

  • /\[bc]at/i 匹配第一个[bc]at,不区分大小写,(元字符需要用\转义)

  • [^adgk] 查找给定集合外的任何字符。/^adc/ 匹配开头的给定集合,$ 匹配结尾的

  • /.at/gi 匹配所有以at结尾的三字符组合,不区分大小写

Iframe有哪些缺点？

  • iframe会阻塞主页面的Onload事件；搜索引擎的检索程序无法解读这种页面，不利于SEO；iframe和主页面共享连接池，而浏览器对相同域的连接有限制，所以会影响页面的并行加载；使用iframe之前需要考虑这两个缺点。如果需要使用iframe，最好是通过javascript动态给iframe添加src属性值，这样可以绕开以上两个问题。
