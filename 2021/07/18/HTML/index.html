错题

  • （规范）ul里面只能放li，但li里可以放h* 和p等标签，dl里可以放dt和dd，dt和dd同级

  • <fieldset> 标签 : 将表单内容分组打包,生成一组相关表单的字段,特殊边框,

    <legend> 标签 : 为filedset元素定义标题

    <p>标签里不能放<p>标签和块级元素标签

  • 不可继承的属性 :

    ① display ② margin ③ border ④ padding

    ⑤ background ⑥ height/width ⑦ position

Html结构

  • HTML: (Hyper Text Markup Language)超文本标记语言,用于构建网页基本结构及其内容
  • 超文本: 文本中指向其他文本的链接
  • 标记语言: 将文本及其他信息结合起来,展现出文档结构,如Html/Xml/Markdown
  • !DOCTYPE 标签: 他是指示web浏览器关于页面使用哪个HTML版本进行编写的指令

HTML5: 主要解决的问题

  1. 文档结构的混乱: 以前的文档结构过度依赖div标签,Html5则提出了多种语义化标签,使得文档易理解,并有利于seo优化
  2. 解决了浏览器之间的兼容性问题
  3. 提供了一些新的web应用的api

  • :刚html文档最前面位置,加上后就会按照W3C的HTML5标准啦解析渲染页面

  • <head> :对用户不可见,包括字符编码声明(utf-8 : 表示页面可用任何语言)、关键字、css样式、页面描述等等

块级元素

  1. 能够设置宽高，默认和父容器宽度一样
  2. 独立成行，(块级元素们)在页面从上往下排列，margin和padding均有效
  3. 可容纳其他块级元素、或内联元素
  4. 代表性的就是div，其他如p、h1-h6、nav、aside、header、footer、section、article、ol、ul-li、dl、dt、dd等等，都可以用div来实现。不过为了可以方便程序员解读代码，一般都会使用特定的语义化标签，使得代码可读性强，且便于查错。
  5. ul和ol里只能放li, li里再放其他 , 如果一个
  6. 元素之后紧跟另一个
  7. 元素，或在父元素中没有其他更多内容，则结束标签可以省略 dd在dt后面但与dt同级

行内(内联)元素

  1. 只占据它对应标签的边框所包含的空间,(不会自动换行) ( 非独立成行 )
  2. 只能容纳文本或其他内联元素
  3. 设置宽高无效
  4. 对margin仅设置左右方向有效，上下无效；padding设置上下左右都有效，
  5. 常见的行内元素:a、span、br、i、em、label( 在 label 元素内点击文本，就会触发for属性绑定的id控件 )、var、strong
  6. a标签的href属性里的URL不加协议，就会变相对路径，浏览器找不到会帮你加协议,a标签的伪类的先后顺序,LV好烦啊: link,visited,hover,focus(点击刚松开,即聚焦,内容是纯文本则无效),active(点击未松开)

行内块级元素

  1. 非独立成行
  2. 能设置宽高
  3. 在行内从左往右排列
  4. 常见行内块元素 ： img、input
  5. 空元素 : 无须闭合的标签，如：<br/>、<hr/>、<input/>、<img/>、<link/>、<meta/> 等 （厉害吗b）
  • 块级元素与内联元素区别： （ 块级元素）单独成行，可设置高度、宽度和上下外边距，

盒模型

（div、article、section）

① 标准盒模型 box-sizing : content-box (默认)

​ 此时盒子的宽和高只决定元素的content部分,即 width = content,加paddding会外撑,即content部分不会变化

② IE(怪异)盒模型 box-sizing : border-box

​ 此时盒子的宽和高决定元素的border大小,即 width = content+padding*2+border*2,加padding会内缩,即border部分不会变化

ps:

  1. scrollHeight=高度height(滚动内容)+上下padding
  2. clientHeight=高度height(滚动容器)+上下padding
  3. offsetWidth = 宽度 + padding + border

H5新特性 (不理玄音,双语本多余)

  1. 新增选择器 document.querySelector、document.querySelectorAll
  2. 音视频API video 和 audio
  3. 本地存储 localStorage(浏览器关闭后数据不丢失)和 sessionStorage
  4. 语义化标签 article、footer、header、nav、section
  5. 多任务 webworker
  6. 全双工通信协议 websocket
  7. 历史管理 history
  8. 跨域资源共享(CORS) Access-Control-Allow-Origin
  9. 画布 canvas ( canvas绘制不操作DOM, svg直接绘制DOM元素 )

html语义化

  • 在编程中，语义指的是一段代码的含义，这个HTML元素有什么作用，扮演了什么样的角色。简单的概括为：在适当的位置使用适当的标签,用正确的标签做正确的事情
  • 脱掉CSS的外衣,依然头是头,脚是脚。还是一篇完整的文档。这就是语义化之后文档的效果

​ 语义化： 也无非就是在使用标签的时候多使用有英文语义的标签。

① header : 头部 ② aside : 独立内容区 ③ main : 主体 ④ footer : 底部 ⑤ section ⑥ article

  • 语义化优点： ( **开胃一姐莉姐 **)
  1. 易于用户阅读，样式丢失的时候能让页面呈现清晰的结构。
  2. 有利于SEO，搜索引擎根据标签来确定上下文和各个关键字的权重。
  3. 方便其他设备解析，如盲人阅读器根据语义渲染网页
  4. 有利于开发和维护，语义化使代码更具可读性、更好维护，与CSS3关系更和谐。

iframe有什么优点、缺点

  • 优点：
  1. iframe能够原封不动的把嵌入的网页展现出来。
  2. 如果有多个网页引用iframe，那么你只需要修改iframe的内容，就可以实现调用的每一个页面内容的更改，方便快捷。
  3. 网页如果为了统一风格，头部和版本都是一样的，就可以写成一个页面，用iframe来嵌套，可以增加代码的可重用。
  4. 如果遇到加载缓慢的第三方内容如图标和广告，这些问题可以由iframe来解决。
  • 缺点：
  1. iframe会阻塞主页面的onload事件；
  2. iframe和主页面共享连接池，而浏览器对相同域的连接有限制，所以会影响页面的并行加载。会产生很多页面，不容易管理。
  3. iframe框架结构有时会让人感到迷惑，如果框架个数多的话，可能会出现上下、左右滚动条，会分散访问者的注意力，用户体验度差。
  4. 代码复杂，无法被一些搜索引擎索引到，这一点很关键，现在的搜索引擎爬虫还不能很好的处理iframe中的内容，所以使用iframe会不利于搜索引擎优化（SEO）。
  5. 很多的移动设备无法完全显示框架，设备兼容性差。
  6. iframe框架页面会增加服务器的http请求，对于大型网站是不可取的。
