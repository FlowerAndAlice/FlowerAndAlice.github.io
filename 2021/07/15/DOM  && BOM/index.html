Web Apis

  • dom: 文档对象模型, 简单理解为处理文档的一些接口

  1. 对于JavaScript，为了能够使JavaScript操作HTML，JavaScript就有了一套自己的DOM编程接口。

  2. 对于HTML，DOM使得html解析后形成一颗DOM树(与CSSOM数组合成Render渲染树,然后布局计算、绘制)，包含文档、元素、节点。

  • 文档: 一个页面就是一个文档,DOM中使用document表示
  • 元素: 页面中的所有标签都是元素,DOM中用element表示
  • 节点: 网页中所有内容都是节点(标签/属性/文本/注释),DOM中用node表示
  • DOM把以上内容都看做对象(可以获取)

DOM操作

  • 我们主要针对元素的操作。主要有创建、增、删、改、查、属性操作、事件操作。

  • 修改样式:

this.style.backgroundColor = “greenyellow”;
this.style.fontSize = “26px”;

  • 创建节点: 通过 let ck =document.createElement(‘up’);
    创建节点之后需要添加节点才能看到创建好的节点
  • 添加节点(添加到末尾): node.appendchild(child);
  • 插入节点(到指定位置的前面): ul.insertBefore(节点绑定名, ul.children[0]);
  • 删除节点(删除并返回一个子节点): ul.removeChild(ul.children[0]);

DOM创建

  • document.write（）是直接将内容写入页面的内容流，但若是文档流执行完毕再进行document.write（），则它会导致页面全部重绘。

  • HTML 输出流：当前数据形式是 HTML 格式的数据，这部分数据正被导出、传输或显示，所以称为”流“。

  • innerHTML是将内容写入某个DOM节点，不会导致页面全部重绘，创建多个元素效率更高（不要拼接字符串，采取数组形式拼接），结构稍微复杂。

  • innerHTML获取标签所包含的HTML+文本信息,innerText只能获取文本信息

  • document.createElement( ) //创建多个元素 效率稍微低一些 但是结构会很清晰

  • document.createTextNode( ) //可创建文本节点,还需后续来插入到元素中

事件模型

  • 比如 : 三个div嵌套,都绑定click事件,点击最内层的元素,事件是从内到外执行的。

  • 事件捕获： 从外层到内层(不会触发事件)

  • 事件处理

  • 事件冒泡： 从内层到外层(会触发事件)
    JS中当触发某些具有冒泡性质的事件时，首先在触发元素寻找是否有相应的注册事件，如果没有再继续向上级父元素寻找是否有相应的注册事件作出相应，这就是事件冒泡。

    • 阻止冒泡: e.stopPropagation(); //此处的e为函数中传的参数

    • 阻止事件的默认行为(比如a链接跳转): e.preventDefault();//一定有效
      或者 return false; //不一定有效

事件委托

  • 利用事件冒泡的特性，将本应该注册在子元素上的处理事件注册在父元素上，这样点击子元素时发现其本身没有相应事件就到父元素上寻找作出响应。比如点击父元素中的子元素,this指向父元素,但e.target指向捕获到底的子元素

  • 这样做的优势：1.减少DOM操作，提高性能。2.随时可以添加子元素，添加的子元素会自动有相应的处理事件。

  • 原因: 在JavaScript中，添加到页面上的事件处理程序数量将直接关系到页面的整体运行性能，因为需要不断的与dom节点进行交互，访问dom的次数越多，引起浏览器重绘与重排的次数也就越多，就会延长整个页面的交互就绪时间，这就是为什么性能优化的主要思想之一就是减少DOM操作的原因；如果要用事件委托，就会将所有的操作放到js程序里面，与dom的操作就只需要交互一次，而且只需要一个内存空间,这样就能大大的减少与dom的交互次数，提高性能；

  • 原理:

  1. Event对象提供了一个属性叫target，可以返回事件的目标节点，成为事件源，也就是说，target就可以表示为当前的事件操作的dom，但是不是真正操作dom;
  2. 此时只是获取了当前节点的位置，并不知道是什么节点名称(但可以直接用)，这里我们可以用nodeName来获取具体是什么标签名，这个返回的是一个大写的，
  3. 用事件委托的方式，新添加的子元素是带有事件效果的，我们可以发现，当用事件委托的时候，根本就不需要去遍历元素的子节点，只需要给父级元素添加事件就好了，其他的都是在js里面的执行，这样可以大大的减少dom操作，这才是事件委托的精髓所在。
  4. 详细请查看:https://www.cnblogs.com/lauzhishuai/p/11263210.html

浏览器显示页面的原理

  • 基本流程 ：
  1. 获取 HTML 文档
  2. 加载外部样式表及JavaScript脚本
  3. 解析执行JavaScript脚本
  4. 解析成对应的树形数据结构
    • DOM tree ( 构建完就是dom ready )
    • CSSOM tree
  5. 计算可见节点形成 render tree
  6. 计算 DOM 的形状及位置进行布局
  7. 将每个节点转化为实际像素绘制到视口上（栅格化）

  • render tree（页面上所显示的最终结果）是由 DOM tree（开发工具中所显示的 HTML 所定义的内容结构）与 CSSOM tree（样式表所定义的规则结构）合并并剔除不可见的节点所形成的，其中不包含如下节点：<html> <head> <meta> <link> <style> <script> 以及设置的 display:none 样式的

  • render tree：渲染树，也就是浏览器引擎通过DOM Tree和CSS Rule Tree构建出来的一个树状结构，和dom tree不一样的是，它只有要最终呈现出来的内容，像或者带有display:none的节点是不存在render tree中的。

  • reflow / layout ( 回流,重排,渲染中的一种行为 )

    • render 树中的一部分(或全部)因为元素的规模尺寸，布局，隐藏等改变而需要重新构建。(一定时间内只会触发一次), 浏览器会使渲染树中受到影响的部分失效，并重新构造这部分渲染树，完成回流后，浏览器会重新绘制受影响的部分到屏幕中，该过程称为重排。

    • display : none ,不再占据空间,DOM结构变化,需重新渲染

    • visibility : hidden, 依旧占据空间,DOM结构未变,无需重新渲染

  • 会导致reflow(回流)的操作

    • 页面首次渲染
    • 浏览器窗口大小发生改变
    • 元素尺寸或位置发生改变
    • 元素内容变化(文字数量或图片大小)
    • 元素字体的大小变化
    • 添加或删除可见的DOM元素 appendChild
    • 激活CSS伪类, (例如 : hover)
    • 查询某些属性或调用某些方法 offsetHeight

  • repaint：重绘，渲染中的一种行为。render tree中任一元素样式属性（几何尺寸没改变）发生改变了，render tree都会重新画，比如字体颜色、背景等变化。

  • reflow一定引起repaint,而repaint不一定引起reflow

  • 资源加载：

    • 执行 JavaScript 会阻止 DOM tree 构建
    • 加载 CSS 会阻止 render tree 构建
      • 无论是否为阻止呈现的 CSS，都会被浏览器下载
    • 默认情况下，JavaScript 脚本会在引入它的位置执行（如果是外联脚本则还需要等待加载完毕），这时会阻断 DOM tree 的构建；如果在运行脚本时浏览器尚未完成 CSS 的下载和 CSSOM tree 的构建，浏览器会将脚本执行延迟到这些操作结束之后。

  • 影响性能的因素

    • 白屏时间( first Paint Time ) : 用户打开页面到页面开始有东西呈现为止

      • HTML 和 CSS 的加载及解析速度

      • <head> 内的脚本加载及执行

    • 首屏时间 : 用户浏览器首屏内所有内容都呈现出来所花费的时间

      • 图片加载
      • \<body> 内的脚本加载及执行

    • render tree 的构建

      • HTML 的复杂度
      • CSS 的复杂度

    • render tree 的绘制（栅格化）

      • 颜色的复杂度
      • 形状的复杂度

  • 用户可操作时间( DOM Interactive ) : 用户可以进行正常地点击、输入等操作，默认可以统计

各步骤的意义如下：

  1. domLoading 表示开始解析第一批收到的 HTML 文档的字节
  2. domInteractive 表示完成全部 HTML 的解析并且 DOM 构建完毕
  3. domContentLoaded 表示 DOM 与 CSSOM 皆已准备就绪
    • 如果没有解析器阻塞 JavaScript，DOMContentLoaded 事件会在 domInteractive 之后立即触发
    • 很多 JavaScript 框架会在执行它们自己的逻辑前等待这个事件的触发
    • DOMContentLoaded 事件必须等待其所属script之前的样式表加载解析完成才会触发。
  4. domComplete 表示所有的处理都已完成并且所有的附属资源都已经下载完毕
  5. loadEvent 作为网页加载的最后一步以便触发附加的应用逻辑

  • readystatechange : document有readyState属性来描述document的loading状态，readyState的改变会触发readystatechange事件.

    • loading : 文档仍然在加载
    • interactive : 文档结束加载并且被解析，但是像图片，样式，frame之类的子资源仍在加载
    • complete : 文档和子资源已经结束加载，该状态表明将要触发load事件。

  • onpageshow 事件类似于 onload 事件，onload 事件在页面第一次加载时触发， onpageshow 事件在每次加载页面时触发，即 onload 事件在页面从浏览器缓存中读取时不触发。

BOM (浏览器对象模型)

  • window对象( 全局对象 )

  • 子对象

    • screen对象 : 包含有关用户屏幕的信息

    • location对象 : 用于获取当前页面的地址(URL),并把浏览器重定向到新的页面

    • history对象 : 包含浏览器的历史
    • navigator对象 : 包含用户浏览器的相关信息

location

  • 代表浏览器的定位和导航。可以使用 location 来操作 URL 中的各个部分。最常用的有 href 属性，当前访问资源的完整路径。

  • 方法 :

    • assign(url) 载入新的 url，记录浏览记录

    • replace(url) 载入新的 url 不记录浏览记录

    • reload() 重新载入当前页

History对象

  • 属性 :

    • length :返回浏览器历史列表中的URL数量

  • 方法 :

    • back( ) : 加载history列表中的前一个URL

    • forward( ) : 加载history列表中的下一个URL

    • go( ) : 根据当前所处的页面，加载history列表中的某个具体页面,正数向前
